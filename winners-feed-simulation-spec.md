# Симуляция активности игроков (лента победителей + банк раунда)

Версия 1.0, 2026-06-11. Для команды разработки (бэкенд + фронтенд).

## 1. Зачем

Из-за задержек мы не можем забирать реальные ставки игроков с 1x в реальном времени,
а пустой экран без активности - продуктовое опущение. На первых итерациях симулируем
правдоподобный поток выигрышей: лента победителей (тикер), общий банк раунда на экране
результата, касса за день.

Главное архитектурное требование: симуляция - это временный источник данных, а не
фича фронтенда. Фронт ничего не знает про симуляцию.

## 2. Архитектура

### 2.1. Интерфейс провайдера

В Game Loop (бэкенд) вводим интерфейс WinnerFeedProvider с двумя реализациями:

- SimulatedFeedProvider - эта спека;
- OperatorFeedProvider - будущая, на реальных данных 1x (вопрос 3.2 требований).

Обе реализации эмитят ОДИНАКОВЫЕ события в шину оверлея. Когда 1x даст данные,
меняем провайдера конфигом сессии, фронт не трогаем. Гибрид (реал с дозаполнением
симуляцией) в MVP не делаем - риск двойного показа.

### 2.2. Где исполняется

Только на бэкенде (внутри Game Loop сервиса). Причины:

- оверлей рендерится сервером и уходит в Live X как видео - картинка одна для всех;
- песочница блогеров и BO подписаны на ту же шину - все видят одно и то же;
- генерация детерминированная (seed), фронт остается тупым рендером.

### 2.3. События (контракт для фронта)

```
sim_round_activity   // на betting_closed
{ session_id, round, bet_count, bank }            // bank = сумма ставок раунда

sim_winner           // 0..N штук после round_result, с разбросом во времени
{ session_id, round, nick_masked, payout, market_tag, ts }
// market_tag: "РАУНД" | "СЕРИЯ ×2..×5" | "ТОТАЛ" | "ЭКСПРЕСС" | "ФОРА"
// коэффициент НЕ передаем и НЕ показываем (у игроков он сильно динамичный)

sim_day_totals       // после каждого раунда
{ date, kassa_today }                             // сумма bank за день, все сессии
```

Фронт: тикер рисует sim_winner (поштучно, с интервалами из ts - не пачкой),
экран результата - bank последнего раунда, подвал - kassa_today.

## 3. Модель генерации

Все случайности - от seeded PRNG (mulberry32 или аналог).
Seed = hash(session_id + date). Один seed = воспроизводимый поток (нужно для QA).

### 3.1. Объем ставок по раундам (фактор "в начале мало, к концу больше")

```
N_bets(r) = round( A(r) * V * noise(r) ),  минимум 5

A(r)   - кривая аудитории: логистический рост 0.35 -> 1.0 на раундах 1..54
         + бамп +0.25 на раундах 1-2 и 28-29 (длинные окна ставок, открытие таймов)
         + спад *0.9 после раунда 50
V      - конфиг "пик ставок за раунд" (дефолт 120)
noise  - lognormal, сигма 0.25, клип [0.6, 1.6] - раунды не монотонны
```

Дополнительный фактор реалистичности: после раунда с крупным выигрышем объем
следующего раунда умножаем на 0.85 (пул "выдохся") - ломает линейность.

### 3.2. Ставки и киты (фактор "большинство мелкие, 5-10% киты")

- Ставка обычного игрока: lognormal, медиана $8, p95 около $100,
  прижатая к "человеческим" номиналам {1,2,5,10,15,20,25,50,100,200}.
- Киты: вероятность кит-события растет по сессии 2% -> 15%. Если случилось,
  число китов в раунде K ~ Poisson(1.2), то есть иногда 0, иногда до 5.
  Ставка кита: {500, 1000, 2000, 5000}.
- bank(r) = сумма реально насэмпленных ставок (консистентность бесплатно).

### 3.3. Маркеты и крупные выигрыши (фактор "у игроков маркетов больше")

Микс маркетов зависит от позиции раунда:

```
обычный раунд:        85% РАУНД, 10% СЕРИЯ, 5% ФОРА/ТОТАЛ
конец четного раунда: +вес расчетам СЕРИЯ ×2/×4 (закрытие коротких серий)
раунды 27 и 54:       инъекция 1-4 крупных выигрышей (ТОТАЛ, ЭКСПРЕСС,
                      победитель тайма/матча) с выплатой 20-150 медиан
последний раунд:      обязательный "хедлайнер" дня - крупнейшая выплата сессии
```

Скрытые коэффициенты (только для расчета выплат, наружу не идут):
РАУНД 1.55-2.45, ФОРА/ТОТАЛ 1.8-3.5, СЕРИЯ 3-30, ЭКСПРЕСС 5-80 (тяжелый хвост).

Важно: крупные выигрыши на четных раундах и концах таймов - вероятностные,
не детерминированные. Жесткое "каждый четный раунд кит" зритель вычислит за вечер.

### 3.4. Бюджет выплат и инварианты (анти-палево)

```
total_payouts(r) = bank(r) * U(0.55 .. 0.92)      // никогда >= 100% банка
один выигрыш     <= 40% от bank(r)                 // иначе поднять bank, не выплату
показываем top-K + случайную выборку, K = clamp(3 + N_bets/40, 3, 12)
```

Если в раунде сломалась серия побед (по реальной истории раундов!) - бюджет
смещаем в "меньше победителей, крупнее выплаты": зрители видели андердога.
Это связывает симуляцию с реальным ходом матча - сильнейший фактор правдоподобия.

### 3.5. Ники (меняются, но повторяются)

- Пул дня: 400 ников, генерируется по seed(date): словарь имен/слов RU+EN +
  опциональный суффикс цифр ("DINARA", "KOTYA88", "MAGA_05", "BARS777").
- Выбор по Zipf - частые повторы у головы пула, длинный хвост одноразовых.
- 20 "завсегдатаев" с весом x5 - зритель узнает знакомые маски.
- Киты - отдельный подпул из 15 ников: хайроллеры как повторяющиеся персонажи.
- Маскирование: первые 1-2 символа + последний ("K***A", "BA***7"). Коллизии
  масок допустимы - даже полезны.
- Ограничения: одна маска максимум 2 раза за раунд и не подряд в тикере;
  один и тот же кит не выигрывает два раунда подряд.

### 3.6. Тайминг эмиссии

- sim_winner отдаем не пачкой: разброс 0.5-4 сек после round_result,
  крупные - последними (драматургия).
- Во время длинного BETTING OPEN (раунды 1, 28) победителей нет - тикер крутит
  накопленную ленту дня; при холодном старте дня - заранее сгенерированный
  "прематч" (10-15 записей от "ранних рынков").

## 4. Персистентность и отказы

- kassa_today и топ ленты дня - в персистентном сторе (Redis/БД), ключ = дата.
  Перезапуск Game Loop не обнуляет кассу (см. 2.7.9 требований).
- round_aborted / void: все несэмиченные sim_winner раунда отменяются,
  sim_round_activity не ревертим (банк был, выплат не было - честно).
- Несколько сессий в день: касса общая, пул ников общий, seed у каждой сессии свой.

## 5. Подводные камни (чеклист для ревью)

1. Консистентность чисел: сумма видимых выплат < банка; касса = сумма банков.
   Зритель со скриншотами не должен поймать арифметику на лжи.
2. Паттерны: ничего строго периодического (киты, хедлайнеры, объемы) - только
   вероятности. Дневной seed меняет картину между днями.
3. Валюта: RUB-сессии умножают шкалы примерно на 90 и снапятся к рублевым
   номиналам (100/500/1000), иначе "$8" превратится в неправдоподобные 8 руб.
4. Округления: выплаты не должны быть круглыми постоянно ($190, $1,247 - ок;
   десять подряд $1,000 - палево). Ставки круглые, выплаты нет.
5. Синхронность поверхностей: оверлей, песочница, BO слушают одни события.
   Никакой генерации на клиенте.
6. Комплаенс: выигрыши фиктивные - нужен sign-off 1x на такую витрину
   (их бренд, их юрисдикции). В коде и доках слой называем simulation, наружу
   термин не светим. План миграции на реальные данные - часть этой же задачи.
7. Никогда не использовать реальные узнаваемые имена/ники стримеров в пуле.
8. Лента и банк должны переживать TIE resolution (несколько card_drawn на раунд)
   - триггер один: round_result.
9. QA: прогон 100 seed x 54 раунда автотестом на инварианты из 3.4; визуальный
   прогон ускоренной сессии (dev-тайминги) на "ощущение живости".

## 6. Конфиг (тюнинг без релиза)

```
peak_bets_per_round: 120        whale_event_prob: 0.02..0.15 (ramp)
payout_budget: [0.55, 0.92]     whale_pool: 15
nick_pool: 400                  regulars: 20
big_win_rounds: [27, 54]        max_winners_shown: 12
currency: USD | RUB             seed: auto(session_id + date)
```

Все параметры - в конфиге сессии/окружения, чтобы крутить без деплоя
после первых живых прогонов.

## 7. Объем работ (оценка для планирования)

- Бэкенд: модуль-генератор + 3 события в шину + персист кассы. ~3-5 дней с тестами.
- Фронтенд: тикер уже есть; подключить события, дрип-показ, банк на результате. ~1-2 дня.
- Референс поведения тикера/банка: bet-cabinet v0.5 в корне репо (там симуляция
  клиентская и примитивная - использовать только как референс UI, не логики).
